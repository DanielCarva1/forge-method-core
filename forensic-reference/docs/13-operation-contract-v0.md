# Operation Contract v0

## Purpose

Define the first typed contract that every Forge host surface must understand.

`OperationContract` is the boundary between Forge Core and borrowed-shell agents. It tells the host agent what Forge believes is true, what the next safe operation is, what autonomy level applies, what may mutate state, and what must stop.

This contract is agent-facing. Humans experience it through the agent conversation, not by reading raw YAML.

## Design Goals

- preserve rich early human guidance
- keep later mechanical work autonomous when decisions are settled
- prevent host agents from inventing route authority
- keep state mutation explicit and narrow
- work through CLI JSON and MCP with identical semantics
- be small enough for agent context
- be schema-validatable by Rust/Serde

## Non-Goals

- define the full state file format
- define all workflow schemas
- define UI copy
- implement a model provider layer
- preserve Python runtime shape

## Top-Level Fields

### `schema_version`

Version of the contract schema. Breaking changes require a new version and migration rule.

### `contract_id`

Unique id for audit, replay, and host logs.

### `created_at`

UTC timestamp generated by Forge Core.

### `project_ref`

The project identity and state version used to create the contract. Hosts must include this when calling a write operation so Forge can detect stale-chat writes.

### `source`

The host, surface, operation, and input digest that produced the contract. It helps diagnose drift and replay failures without storing full chat text in every contract.

## Autonomy

```yaml
autonomy:
  mode: "observe|facilitate|research|plan|execute|repair|gate_review|diagnose"
  rationale: "short reason"
```

### Modes

`observe`
: Read state and explain. No mutation.

`facilitate`
: Ask useful human questions before committing direction.

`research`
: Gather external or internal evidence before decisions.

`plan`
: Close decisions into contracts, artifacts, or stories.

`execute`
: Perform mechanical work from accepted contracts.

`repair`
: Correct route, state, artifact, install, or workflow drift.

`gate_review`
: Validate readiness, quality, authority, evidence, or release.

`diagnose`
: Inspect host/runtime/install/version health.

## Recommendation

```yaml
recommendation:
  next_actor: "human|host_agent|forge_core"
  next_operation: "guide|oracle|gate|doctor|apply_transition|record_artifact|record_evidence|record_decision|record_request|claim_lane|release_lane|update_heartbeat|null"
  host_action: "ask_human|show_status|call_operation|run_command|stop"
  phase: "1-discovery"
  workflow: "product-direction"
  action: "ask_first_question"
```

`next_operation` cannot be `advance`.

When `next_actor` is `human`, `next_operation` must be `null` and `host_action` must tell the host agent how to interact with the human.

When `next_actor` is `host_agent`, `next_operation` may be `null` for host-local action such as `show_status` or `run_command`, or it may name the Forge read operation the host should call.

When `next_actor` is `forge_core`, `host_action` should be `call_operation`.

There is no generic `write` operation. Mutating recommendations must name the exact write operation.

## Authority

```yaml
authority:
  mutation_policy: "forbidden|requires_human|requires_gate|allowed"
  side_effect_policy: "read_only|ask_before_write|write_project_files|run_safe_commands|run_external_commands|system_update"
  authority_sources:
    - "human_explicit"
    - "workflow_gate"
    - "state_machine"
    - "lane_claim"
    - "claim_contract"
    - "completion_contract"
    - "system_maintenance"
  authority_evidence:
    - kind: "gate_result|human_input|state_rule|lane_claim|claim_contract|completion_contract|system_policy"
      ref: "short stable reference"
  missing_authority: []
```

The host agent may reason broadly, but it may not mutate Forge state unless `mutation_policy` and `authority_sources` allow it.

`operation_contract` is not an authority source. The contract is the envelope that carries authority evidence. This prevents a malformed or host-influenced contract from becoming self-authorizing.

## Coordination Scope

`coordination_scope` defines the target and coordination proof for the next operation. It replaces a weaker `write_scope` concept because safe mutation is not only about where the agent writes; it is also about who may write, under what version, claim, lane, and completion state.

```yaml
coordination_scope:
  target:
    kind: "none|project|product_area|story|artifact|lane|evidence|integration_state"
    id: "stable-id-or-null"
    product_area: "runtime-core"
    paths:
      - "crates/forge-core"
  concurrency:
    expected_state_version: 42
    agent_id: "codex-main"
    caller_role: "driver"
    fleet_mode: true
    registry_ref: ".forge-method/agents/registry.yaml"
  write_authority:
    requires_driver_claim: false
    requires_lane_claim: true
    claim_contract_ref: "contracts/claims/story-v2-010-active-claim.yaml"
  completion:
    must_check_completion: true
    completion_contract_ref: "contracts/completion/story-current-completion.yaml"
```

Rules:

- Mutating operations must include `coordination_scope`.
- In fleet mode, mutating operations require `agent_id`.
- In fleet mode, mutating operations require `caller_role`.
- Integration state mutation requires driver claim evidence.
- Product Area, story, artifact, and lane writes require lane claim evidence unless the operation is `claim_lane`.
- Claim evidence must be `ClaimContract`, not an untyped lock file.
- Expired claims must route to handoff/recovery; they must not silently release work.
- Worker attempts to change integration state must route to `record_request`, not `apply_transition`.
- Build/story recommendations must check completion state before claiming or writing.
- Mutating operations that edit files, state, artifacts, evidence, requests, or git staging must reference `effect_contract_refs`.
- Single-agent mode may set claim fields to `false`, but still carries `expected_state_version`.

## Tool Effect Contract

`ToolEffectContract` covers fine-grained read/write conflict behavior for mutating operations. `OperationContract` decides the next safe operation; `ToolEffectContract` describes the read and write sets that operation may touch.

```yaml
effect_contract_refs:
  - "contracts/effects/story-artifact-write-effect.yaml"
```

Rules:

- `read_set` and `write_set` live in `ToolEffectContract`, not as loose `OperationContract` fields.
- Every write target must have an expected hash, expected version, append-only policy, or explicit driver/human authority.
- If a write touches a changed read target, the default v0 strategy is `notify_and_repair`, not blind abort/retry.
- Destructive writes require inverse/compensation metadata or must stop.

## Request Contract

`record_request` appends a typed worker-to-driver request. It is used when an agent should not mutate integration state directly but has a proposed state change, dependency, handoff, or integration request.

```yaml
request_contract:
  id: "req_..."
  kind: "state_transition|integration_apply|handoff|dependency|question|review_request|conflict_notification|claim_expiry_handoff"
  sender_agent_id: "cursor-worker-1"
  sender_role: "worker"
  target_driver: "codex-main"
  requested_operation: "apply_transition"
  target:
    kind: "integration_state"
    id: "story-ready"
  reason_code: "story_ready_for_integration"
  evidence_refs:
    - ".forge-method/evidence/story-validation.json"
  response_required: true
  deadline: null
  status: "pending"
```

Requests must be structured messages, not freeform handoff prose. The human-facing explanation may be conversational, but the stored request is typed.

## Execution Policy

`execution_policy` defines deterministic control flow for the host. The host agent may reason inside these limits, but it must not invent loops, retries, branches, or continuation policy.

```yaml
execution_policy:
  mode: "observe_only|single_step|bounded_loop|autonomous_sequence"
  max_steps: 1
  retry_policy:
    max_attempts: 0
    on_failure: "stop|ask_human|run_gate|record_request"
  branch_policy:
    allowed_branches:
      - "repair"
      - "gate"
    default_branch: "stop"
```

Mode meanings:

- `observe_only`: explain, ask, or inspect without mutation.
- `single_step`: call exactly one Forge operation or command ref.
- `bounded_loop`: repeat a narrow operation until a stop condition or max step limit.
- `autonomous_sequence`: perform a known sequence of implementation/validation/evidence steps within the contract's scope.

Funnel Autonomy is expressed through this policy:

- early discovery and correct-course normally use `observe_only`;
- planning/gate steps use `single_step`;
- story/test repair may use `bounded_loop`;
- locked mechanical build may use `autonomous_sequence`;
- release and production-sensitive work return to `single_step` or gate-heavy policies.

## Stop Policy

`stop_policy` defines when the host must stop and who acts next.

```yaml
stop_policy:
  stop_when:
    - "human_input_required"
    - "missing_authority"
  on_stop:
    next_actor: "human"
    next_operation: null
    host_action: "ask_human"
```

The stop policy is authoritative. A host must not continue, retry, branch, or mutate after a stop condition unless a new `OperationContract` authorizes it.

## Allowed and Forbidden Actions

These lists are intentionally explicit. They are cheap guardrails for host agents and conformance tests.

Examples:

```yaml
allowed_actions:
  - "ask_human"
  - "read_contract"
  - "write_project_file"
forbidden_actions:
  - "change_route_without_contract"
  - "write_checkpoint_from_suggestion"
  - "mark_ready_without_gate"
```

## Human Prompt

```yaml
human:
  input_requirement: "none|now|before_mutation|before_completion|checkpoint"
  prompt:
    mode: "question|status|choice|challenge|none"
    text: "short human-facing prompt"
    options:
      - id: "fast"
        label: "Fast path"
        tradeoff: "Less exploration."
  tone_contract: "curious_direct"
```

This is not the full human conversation script. It is the compact behavior instruction the host agent uses to speak naturally.

`input_requirement` is separate from `recommendation.next_actor`.

- `none`: no human input is needed for the current contract.
- `now`: the next actor should be the human.
- `before_mutation`: Forge may run read-only checks first, but cannot mutate state before human input.
- `before_completion`: autonomous work may proceed, but completion requires a human checkpoint.
- `checkpoint`: the host should show status and pause at a named review boundary.

## Loads

```yaml
loads:
  required:
    - "contracts/workflows/story-build.yaml"
  optional:
    - ".forge-method/evidence/latest.json"
```

The contract tells the agent what to load next. The agent should not load large Markdown docs by default.

## Gates

```yaml
gates:
  required_before_mutation:
    - scope: "lane"
      gate_contract_ref: "contracts/gates/story-ready-lane-gate.yaml"
  current_gate_status: "pass|fail|concerns|missing|not_applicable"
  gate_contract_refs:
    - "contracts/gates/story-ready-lane-gate.yaml"
```

Gate status is read-only in this contract. Gate decisions are produced by `gate`.

Gate scope is authority scope:

- `lane`: validates local story/lane work only.
- `product_area`: validates a Product Area boundary only.
- `integration`: validates cross-lane or project-wide integration.
- `release`: validates release readiness and publishing.

A gate pass never promotes itself to a broader scope. Lane and Product Area gates may feed an integration gate, but cannot replace it. Integration gates may feed a release gate, but cannot replace it.

## Decision Close

Decision-closing mutations require `DecisionCloseContract` evidence. This covers phase transitions, handoff, spec lock, route change, and release readiness.

```yaml
decision_close:
  ref: "contracts/decisions/phase-transition-missing-grill.yaml"
```

Rules:

- Decision close requires grill evidence in v0.
- Missing grill evidence blocks `apply_transition`, `record_decision`, spec lock, handoff close, and release readiness.
- Ordinary mechanical story execution is governed by scope, claims, gates, completion, and effects; it does not need decision close evidence unless it closes a decision.

## Command References

Operation contracts do not carry raw shell commands. They reference typed `CommandContract` records.

```yaml
command_refs:
  - id: "cmd.validate.story_fast"
    required: true
```

Commands that mutate files, install packages, publish releases, or call external systems require compatible `side_effect_policy` in both the `OperationContract` and the referenced `CommandContract`.

`CommandContract` must model executor, args, cwd policy, platform support, timeout, environment policy, network policy, and side-effect policy as typed fields. Shell strings are not accepted as the canonical command representation.

## Write Operation Contracts

Every mutating operation receives the `OperationContract` id plus a narrow payload.

Examples:

```yaml
apply_transition:
  contract_id: "op_..."
  expected_state_version: 42
  transition:
    phase: "2-specification"
    workflow: "product-direction"
    status: "direction-approved"
```

```yaml
record_evidence:
  contract_id: "op_..."
  expected_state_version: 43
  evidence:
    kind: "test"
    path: ".forge-method/evidence/20260624-contract-tests.json"
    result: "pass"
```

## Host Conformance Rules

Hosts must:

- preserve `mutation_policy`
- preserve `side_effect_policy`
- preserve `authority_sources`
- preserve `authority_evidence`
- refuse write calls with stale `state_version`
- refuse generic continuation as a write operation
- call the named write operation instead of inventing a route
- resolve only typed `command_refs`
- show or paraphrase `human.prompt` when human input is required
- stop instead of re-calling `guide` when `next_actor` is `human`
- keep direct `oracle`, `gate`, and `doctor` calls read-only
